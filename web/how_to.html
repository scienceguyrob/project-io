Setup & Usage Guide

How to Use the Container

Everything you need to get the notebook environment running, connect to it from your browser, and move files in and out — on macOS or Windows.

Note: This guide covers command-line Docker only. It assumes you have Docker installed. If you do not have Docker yet, visit docs.docker.com/get-docker and follow the instructions for your operating system before continuing.

1. What is Docker?

Docker is a tool that packages software — along with everything it needs to run — into a self-contained unit called a container. Think of it as a lightweight virtual machine that includes a specific version of Python and all the required libraries, pre-installed and pre-configured.

The benefit for you is that the environment is identical for everyone, regardless of what operating system or Python version you have on your own machine. You do not need to install Python, pip, or any libraries yourself. Everything is inside the container.

Key terms you will see:

  • Image — the blueprint for the container, downloaded once from a registry.
  • Container — a running instance of an image. You can start and stop it as many times as you like.
  • Volume — a way of linking a folder on your machine to a folder inside the container, so files can pass between them.
  • Port — a communication channel. JupyterLab runs on a port inside the container; you map it to a port on your machine so your browser can reach it.

2. Pulling the Image

Before you can run a container you need to download the image. Open a terminal (macOS: Terminal or iTerm; Windows: Command Prompt or PowerShell) and run:

docker pull project-io

Replace <image-name> with the image name you have been given. This downloads the image to your machine. You only need to do this once (or again later if an updated version is released).

3. Starting the Container

The command below starts the container and maps JupyterLab to port 8888 on your machine. Run it from a terminal.

macOS

docker run -p 8888:8888 project-io

Windows (Command Prompt or PowerShell)

docker run -p 8888:8888 project-io

The command is the same on both platforms. After a few seconds you will see output from JupyterLab in your terminal. Leave this terminal open — closing it stops the container.

What -p 8888:8888 means: The first 8888 is the port on your machine; the second is the port inside the container. When you visit http://localhost:8888 in your browser, your machine forwards that request to JupyterLab running inside the container.

4. Connecting to JupyterLab

Once the container is running, open your browser and go to:

http://localhost:8888

JupyterLab will open. No password or token is required. You will see the file browser on the left showing the notebooks and folders described on the Home page.

To open a notebook, navigate to its folder in the file browser and double-click the .ipynb file.

Recommended: open HTML pages directly in the browser.
The HTML pages in this series (this guide, the home page, and the notebook search index) are best viewed as standalone browser pages rather than through JupyterLab's file viewer, which applies restrictions that can prevent interactive features from working.

To open any HTML file directly, use this URL pattern in your browser:

http://localhost:8888/files/index.html
http://localhost:8888/files/web/search.html
http://localhost:8888/files/web/challenges.html
http://localhost:8888/files/web/data.html
http://localhost:8888/files/web/how_to.html

Simply type or paste the address into your browser's address bar.

5. Stopping the Container

To stop the container, go to the terminal where it is running and press Ctrl + C. The container will shut down cleanly.

If you started the container in the background (with the -d flag), stop it by name or ID:

docker ps                        # lists running containers and their IDs
docker stop <container-id>       # stops the container gracefully

6. Moving Files In and Out of the Container

There are two ways to share files between your machine and the container: volume mounts (recommended) and docker cp.

Option A — Volume Mount (recommended)

A volume mount links a folder on your machine to a folder inside the container. Files in that folder are shared in real time — changes on either side appear immediately on the other. This is the cleanest approach for saving your work.

Add the -v flag to your docker run command:

macOS

docker run -p 8888:8888 -v /Users/yourname/ml-work:/home/jovyan/work <image-name>

Windows (Command Prompt)

docker run -p 8888:8888 -v C:\Users\yourname\ml-work:/home/jovyan/work <image-name>

Windows (PowerShell)

docker run -p 8888:8888 -v C:/Users/yourname/ml-work:/home/jovyan/work <image-name>

Replace /Users/yourname/ml-work (macOS) or C:\Users\yourname\ml-work (Windows) with the actual path to a folder on your machine. Inside the container, the folder will appear at /home/jovyan/work and will be visible in the JupyterLab file browser under the work directory.

Any file you save into that work folder from inside JupyterLab will immediately appear in the folder on your machine, and vice versa.

Windows note: Docker on Windows requires that the drive you are sharing (e.g. C:) is enabled for file sharing in Docker's settings. If you get a permission error, check your Docker configuration.

Option B — docker cp

If you need to copy a specific file rather than mounting a whole folder, use docker cp. This works while the container is running.

Copy a file from your machine into the container:

docker cp /path/to/myfile.csv <container-id>:/home/jovyan/myfile.csv

Copy a file from the container to your machine:

docker cp <container-id>:/home/jovyan/myfile.csv /path/to/destination/

To find the container ID, run docker ps in a separate terminal while the container is running. The ID is the first column of the output.

7. Keeping Your Work Between Sessions

By default, any changes you make to files inside the container are lost when the container stops. To preserve your work across sessions, use a volume mount (Option A above) and save your notebooks and output files into the mounted folder.

Alternatively, use docker cp to copy important files out of the container before stopping it.

8. Opening a Notebook Directly

Rather than navigating the file browser every time, you can open a specific notebook directly by typing its path into your browser after the JupyterLab URL. The pattern is:

http://localhost:8888/lab/tree/Notebook_3/Notebook_3.ipynb

Adjust the path for whichever notebook you want. You can bookmark these URLs for fast access.
